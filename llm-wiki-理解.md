# LLM Wiki 模式理解

## 核心理念

LLM Wiki 是一种革命性的个人知识管理方法，与传统的 RAG（检索增强生成）系统有本质区别：

### 传统 RAG 的局限
- 每次查询都从原始文档重新检索和推导知识
- 没有知识积累和沉淀
- 复杂问题需要重复整合多个文档片段

### LLM Wiki 的创新
- **持久化知识编译**：LLM 持续构建和维护一个结构化的 wiki 系统
- **知识复利增长**：每次添加新资源都会丰富整个知识网络
- **零维护成本**：LLM 承担所有繁琐的书目整理工作

## 架构设计

### 三层结构

```
┌─────────────────────────────────────┐
│   Schema (CLAUDE.md/AGENTS.md)      │  配置层：指导 LLM 如何运作
├─────────────────────────────────────┤
│   Wiki (Markdown Files)             │  知识层：LLM 维护的结构化内容
├─────────────────────────────────────┤
│   Raw Sources (Immutable)           │  源数据层：原始文档集合
└─────────────────────────────────────┘
```

**关键特性**：
- 原始资源不可变，作为事实来源
- Wiki 完全由 LLM 拥有和维护
- Schema 定义工作流程和约定

## 核心操作流程

### 1. 摄入 (Ingest)
添加新资源 → LLM 阅读 → 与用户讨论 → 写入摘要 → 更新索引和实体页面 → 记录日志

**特点**：
- 一个资源可能影响 10-15 个 wiki 页面
- 推荐逐个摄入并保持人工参与
- 支持批量摄入模式

### 2. 查询 (Query)
读取索引 → 定位相关页面 → 综合回答 → **有价值的答案可存回 wiki**

**洞察**：探索过程产生的洞见也应该沉淀到知识库中

### 3. 检查 (Lint)
定期健康检查：
- 页面间的矛盾
- 过时的信息
- 孤立页面（无引用）
- 缺失的交叉引用
- 可通过搜索填补的数据空白

## 导航机制

### index.md（内容目录）
- 按类别组织所有页面
- 每页包含：链接、摘要、可选元数据
- 摄入时自动更新
- 支持中等规模（~100 源文档，数百页面）

### log.md（操作日志）
- 仅追加的时间顺序记录
- 统一前缀格式便于解析
- 示例：`## [2026-04-02] ingest | Article Title`
- 可用 Unix 工具快速查询历史

## 应用场景

### 个人层面
- 目标追踪、健康管理、自我提升
- 整合日记、文章、播客笔记

### 研究领域
- 深度主题研究（论文、报告、文章）
- 构建逐步完善的论文 wiki

### 阅读学习
- 逐章整理书籍内容
- 构建角色、主题、情节关系网
- 类似粉丝 wiki（如 Tolkien Gateway）

### 团队协作
- 内部知识库维护
- 自动整合 Slack、会议记录、项目文档
- 解决"无人愿意维护文档"的困境

### 其他场景
- 竞品分析、尽职调查
- 旅行规划、课程笔记
- 任何需要长期积累知识的领域

## 工具生态

### 必备工具
- **Obsidian**：阅读和浏览 wiki 的 IDE
- **LLM Agent**：编写和维护 wiki 的程序员
- **Git**：版本控制和协作

### 增强工具
- **Obsidian Web Clipper**：网页转 markdown
- **qmd**：本地 markdown 搜索引擎（支持混合 BM25/向量搜索）
- **Marp**：从 wiki 生成幻灯片
- **Dataview**：基于 YAML frontmatter 的动态查询

### 图像处理技巧
- 将附件存储在固定目录（如 `raw/assets/`）
- 使用热键批量下载图片
- LLM 先读文本，再单独查看图片获取上下文

## 核心价值

### 解决痛点
传统知识库失败的原因：**维护负担增长快于价值**

LLM Wiki 的解决方案：
- LLM 不感到厌倦
- 不会忘记更新交叉引用
- 单次操作可触及多个文件
- **维护成本接近零**

### 人类与 LLM 的分工

| 人类 | LLM |
|------|-----|
| 策划资源 | 总结内容 |
| 指导分析 | 交叉引用 |
| 提出问题 | 文档整理 |
| 思考意义 | 保持一致性 |

## 历史渊源

这与 Vannevar Bush 1945 年提出的 Memex 概念一脉相承：
- 个人策划的知识存储
- 文档间的关联路径
- **解决了 Bush 无法解决的问题：谁来维护？**

## 实施建议

1. **渐进式采用**：从简单场景开始
2. **与 LLM 共同演化**：Schema 文件应在使用中优化
3. **模块化选择**：根据需求挑选工具和方法
4. **保持抽象性**：本模式描述理念，具体实现因人而异

## 我的思考

### 优势
- 彻底改变知识管理的工作流
- 让个人知识库真正可持续
- 实现知识的复利增长
- 符合"编码思维"：wiki 即代码库

### 潜在挑战
- 需要稳定的 LLM 服务
- 依赖良好的 Schema 设计
- 大规模 wiki 可能需要搜索基础设施
- 初期需要投入时间设计工作流

### 未来可能性
- 多人协作的 wiki 系统
- 与专业工具的深度集成
- 自动化资源发现和摄入
- 更智能的矛盾检测和解决方案

---

**总结**：LLM Wiki 模式是一个范式转变——从"每次查询重新推导"到"持续编译和累积知识"。它让个人知识库的长期维护成为可能，是知识工作者与 AI 协作的典范模式。
